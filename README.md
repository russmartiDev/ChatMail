# ChatMail AI Backend

Node.js + Express backend for ChatMail AI. Handles Gmail OAuth, email fetching, and AI processing.

## Setup

### 1. Install dependencies
```bash
npm install
```

### 2. Configure environment
```bash
cp .env.example .env
```
Then fill in `.env`:
- `JWT_SECRET` — any long random string
- `ANTHROPIC_API_KEY` — from https://console.anthropic.com
- `GOOGLE_CLIENT_ID` + `GOOGLE_CLIENT_SECRET` — from https://console.cloud.google.com
  - Enable Gmail API
  - Create **Web Application** OAuth 2.0 credentials
  - Add `http://localhost:3000/auth/gmail/callback` as Authorized Redirect URI

#### Database configuration
This backend requires a PostgreSQL database; SQLite support has been dropped.
Set the `DATABASE_URL` environment variable with a full connection string.
The application will refuse to start if the value is missing.

Example connection string:
```

```

(the previous migration utilities are no longer applicable once SQLite was
removed.)

### 3. Run
```bash
# Development (auto-restart)
npm run dev

# Production
npm start
```

### 4. Deploying to Google Cloud Run
Cloud Run will build your Dockerfile automatically when you connect the
GitHub repo and enable Cloud Build triggers. Ensure you have a `Dockerfile`
(and `.dockerignore`) at the repo root (this project uses the one included
with the source).

A sample `cloudbuild.yaml` is provided which builds the image,
pushes it to Container Registry, and deploys the service. Environment
variables are supplied via the `--set-env-vars` flag or using Secret Manager.

Example manual deploy command:
```bash
# set variables in your shell or use Secret Manager substitutions
gcloud run deploy mailmind-backend \
    --image gcr.io/$PROJECT_ID/mailmind-backend \
    --region us-central1 \
    --platform managed \
    --allow-unauthenticated \
    --set-env-vars "PORT=8080,JWT_SECRET=$JWT_SECRET,ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY,GEMINI_API_KEY=$GEMINI_API_KEY,GOOGLE_CLIENT_ID=$GOOGLE_CLIENT_ID,GOOGLE_CLIENT_SECRET=$GOOGLE_CLIENT_SECRET,GOOGLE_REDIRECT_URI=$GOOGLE_REDIRECT_URI,ALLOWED_ORIGINS=$ALLOWED_ORIGINS,DATABASE_URL=$DATABASE_URL"
```

Once deployed, Cloud Run will automatically restart the service whenever you
push to the configured branch (e.g. `main`).


## Database & Embeddings

A new `email_vectors` table holds embeddings for RAG search.  It's defined
using Postgres's `vector` extension and looks like:

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE email_vectors (
    id TEXT PRIMARY KEY,
    content TEXT NOT NULL,
    metadata JSONB,
    embedding VECTOR(3072) NOT NULL
);
```

The `content` field is generated by Gemini from each email (subject, sender,
body) to produce a searchable summary; this makes the vectors more useful in a
retrieval‑augmented search scenario.

To recreate and populate the table, run:

```bash
cd mailmind-backend
node scripts/embed_emails.js
```

The script automatically installs the `vector` extension, drops the old
`email_vectors` table, and creates it with correct column types; it then
loops over all emails, generating a RAG‑friendly summary via Gemini and
storing the resulting vector (formatted as `[v1,v2,…]`).  **Every execution
starts from scratch**, so run it only when you're ready to rebuild the index
– you'll need to adapt it for incremental updates if you prefer not to wipe
data.

## API Endpoints

### Auth
| Method | Path | Description |
|--------|------|-------------|
| POST | `/auth/register` | Create account `{ email, password, name }` |
| POST | `/auth/login` | Login `{ email, password }` → returns JWT |
| GET | `/auth/me` | Get current user (requires JWT) |

### Gmail
| Method | Path | Description |
|--------|------|-------------|
| GET | `/auth/gmail/url` | Get Google OAuth URL to open in browser |
| GET | `/auth/gmail/callback` | OAuth callback (Google redirects here) |
| GET | `/auth/gmail/status` | Check if Gmail is connected |
| DELETE | `/auth/gmail` | Disconnect Gmail |

### Emails
| Method | Path | Description |
|--------|------|-------------|
| POST | `/emails/sync` | Fetch Gmail + AI process → cache result |
| GET | `/emails` | Return cached processed emails |
| DELETE | `/emails` | Clear cached emails |

## Architecture

```
Mobile App
   │
   ├── POST /auth/login → JWT token
   │
   ├── GET /auth/gmail/url → opens browser
   │        └── Google OAuth → /auth/gmail/callback
   │                └── tokens saved to user
   │
   └── POST /emails/sync
            ├── fetchEmails() → Gmail API (using stored tokens)
            └── processEmails() → Claude AI
                     └── returns categorized batch
```

## Troubleshooting

### CORS errors from the mobile app or browser
If your client (Expo, React web, etc.) complains about blocked requests and
you see **“No 'Access-Control-Allow-Origin' header is present”**, the
backend either crashed before applying the CORS middleware or the origin
wasn’t permitted.

Configuration rules:

* **ALLOWED_ORIGINS** – comma‑separated list of prefix strings. An incoming
  request’s `Origin` header is allowed if it starts with one of the prefixes.
* **ALLOW_ALL_ORIGINS** – set to `true` to bypass all origin checking and allow
  every origin (convenient for quick debugging).
* If **ALLOWED_ORIGINS** is empty and **ALLOW_ALL_ORIGINS** is unset, the
  server defaults to allowing any origin.

The server logs every check; look for lines like:

```
CORS check, origin= http://localhost:8081 allowed list=[https://chatmail-…] allowAll= false
```

To permit all origins unconditionally, either

```bash
gcloud run services update chatmail \
    --region asia-southeast1 \
    --update-env-vars "ALLOW_ALL_ORIGINS=true"
```

or leave `ALLOWED_ORIGINS` blank. To limit origins, populate
`ALLOWED_ORIGINS` appropriately (e.g. include `http://localhost:8081` for
local Expo).

You can also adjust these values via the Cloud Run console under **Variables
& Secrets**.


### Container fails to start on Cloud Run
If your revision stalls with a message like:

> The user-provided container failed to start and listen on the port defined
> provided by the PORT=8080 environment variable within the allocated timeout.

it usually means the process crashed before calling `app.listen()`.

Common causes:

1. **Missing required environment variables.**
   - The server now validates its configuration on startup and will print
     `✖ missing required environment variable: NAME` then exit.
   - Check your Cloud Run service settings or your Cloud Build trigger
     substitutions and make sure you’ve supplied `DATABASE_URL`,
     `JWT_SECRET`, API keys, etc.
   - You can view logs with `gcloud run logs read mailmind-backend --region us-central1`
     or via the Cloud Console.

2. **Port configuration.**
   - Cloud Run sets `PORT=8080`. The app respects this value and falls back to
     8080 if it isn’t set (development default is 3000).
   - No changes are required in most cases; just ensure your Dockerfile exposes
     8080 (the provided Dockerfile does).

3. **Health-check timeout.**
   - By default Cloud Run allows 4 minutes; you can extend it with
     `--timeout` on `gcloud run deploy` if your startup takes longer.

After resolving the configuration issue, deploy again – the container should
start normally and listen on the correct port.

## Testing with curl

```bash
# Register
curl -X POST http://localhost:3000/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"password123","name":"Test User"}'

# Login
curl -X POST http://localhost:3000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"password123"}'

# Get Gmail OAuth URL (replace TOKEN)
curl http://localhost:3000/auth/gmail/url \
  -H "Authorization: Bearer TOKEN"

# Sync emails
curl -X POST http://localhost:3000/emails/sync \
  -H "Authorization: Bearer TOKEN"
```
